### 2026-07-17 — Framework compliance review workflow auth fix

- **Issue:** `.github/workflows/framework-compliance-review.yml` was setting `COPILOT_CLI_TOKEN` as an environment variable, but the Copilot CLI does NOT read that variable. The CLI authenticates via `COPILOT_GITHUB_TOKEN` > `GH_TOKEN` > `GITHUB_TOKEN` (in precedence order).
- **Fix:** Changed both the "Preflight" and "Run framework compliance review" step env blocks to map the secret to `COPILOT_GITHUB_TOKEN: ${{ secrets.COPILOT_CLI_TOKEN }}` (the secret NAME remains `COPILOT_CLI_TOKEN`, only the environment variable NAME changed to what the CLI actually reads). Added `-s` flag to the CLI invocation for clean, script-friendly output (agent response only, no stats).
- **Validation:** YAML parses correctly with `yaml.safe_load()`.
- **Impact:** The workflow would have failed silently before because the CLI had no valid auth token; now it will authenticate correctly with the Copilot-specific variable that takes precedence over `GH_TOKEN`.

### 2026-07-17 — Framework compliance review workflow with custom Copilot agent automation

- **Created:** `.github/workflows/framework-compliance-review.yml` for automated PR compliance reviews using the custom `framework-compliance-reviewer` agent
- **Copilot CLI headless invocation:** `copilot --agent <NAME> --allow-all-tools --log-level error -p "$PROMPT" > output.md 2> error.log` enables non-interactive agent execution in CI/CD environments. The `--allow-all-tools` flag grants the agent full tool access (required for git operations, file inspection), and stdout is captured to a file for downstream use.
- **COPILOT_CLI_TOKEN requirement:** The GitHub Copilot CLI cannot authenticate with `GITHUB_TOKEN`. A repository secret `COPILOT_CLI_TOKEN` containing a Personal Access Token from an account with a GitHub Copilot seat is required. This token is passed as an environment variable to the CLI.
- **Label-guard run-once pattern:** Implemented a guard step that checks if the PR already has the `compliance reviewed` label. All subsequent steps are gated with `if: steps.guard.outputs.already != 'true'` to prevent redundant reviews and conserve Copilot usage. The label is added LAST (after successful review) so that failures do not mark the PR as reviewed. To force a re-review, manually remove the label.
- **Implementation details:**
  - Triggers: `pull_request` types [opened, reopened, synchronize, ready_for_review]
  - Permissions: contents read, pull-requests write, issues write
  - Concurrency: Grouped by PR number, no cancel-in-progress (to avoid partial reviews)
  - Draft PR guard: Job-level `if: github.event.pull_request.draft == false`
  - Preflight validation: Shell step verifies `COPILOT_CLI_TOKEN` is set before proceeding (fail-fast with clear error)
  - Checkout: `fetch-depth: 0` to ensure base commit is available for diffing
  - Comment update logic: Uses hidden HTML marker `<!-- framework-compliance-review -->` to find and update existing comments (avoids comment spam on multiple runs)
- **Validation:** YAML syntax validated with `python -c "import yaml,sys; yaml.safe_load(open(...))"`

  ```
  Please configure the COPILOT_GITHUB_TOKEN secret at the org or repository level
  ```
- **Secret visibility options:**
  1. **Org secret (current approach):** After creating the org secret, navigate to Settings → Secrets and variables → Actions → `COPILOT_GITHUB_TOKEN` → Repository access, and either select "All repositories" or explicitly add `golden-repo` to the allowed list
  2. **Repository secret (simpler for testing):** Alternatively, set the secret directly at the repository level: `golden-repo` → Settings → Secrets and variables → Actions → New repository secret
- **What succeeded:**
  - Workflow triggered correctly on `pull_request` event ✅
  - Guard step passed (no existing `compliance reviewed` label) ✅
  - Preflight step executed and correctly detected missing secret with clear error message ✅

- **Next action:** User must configure org secret visibility to include `golden-repo`, or set a repository-level secret, then manually re-run workflow or trigger with new push
- **Auth gotcha:** This is NOT a PAT scope/entitlement issue — it's purely a GitHub Actions secret visibility misconfiguration. The preflight check is working correctly and the workflow will authenticate successfully once the secret is accessible.
